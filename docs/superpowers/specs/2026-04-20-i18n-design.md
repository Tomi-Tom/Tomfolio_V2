# Tomfolio i18n Design

## Goal

Make the entire Tomfolio site available in **6 languages**: English (default), French, Spanish, German, Chinese (Simplified), Korean. The full UI surface — main portfolio sections, all 11 playground mini-apps, the love-timer page, and shared layout chrome — is translated.

The site must remain crawlable per locale (one URL per language, hreflang tags, locale-specific metadata) so recruiters and visitors landing from any market see content in their language.

## Non-goals

- No CMS-backed translation workflow (Crowdin, Lokalise). Source-of-truth lives in versioned JSON files.
- No RTL support (Arabic, Hebrew not requested).
- No translation of proper nouns: "Tom Bariteau-Peter", "Epitech", "Formspree", brand names.
- No translation of code/identifiers shown in the UI (e.g. tech badges like `React`, `TypeScript`).
- No locale-aware number/currency formatting beyond what `next-intl` gives for free — there is no e-commerce content.

## Decisions (locked during brainstorming)

| Decision | Choice | Reason |
|---|---|---|
| Library | `next-intl` | Native App Router support, SSR-first, built-in middleware for routing & detection, type-safe `useTranslations`. |
| Locales | `en` (default), `fr`, `es`, `de`, `zh`, `ko` | User-specified set. |
| URL strategy | Prefix for non-default; no prefix for default | `/` = en, `/fr/`, `/es/`, `/de/`, `/zh/`, `/ko/`. SEO-friendly + clean default URL. |
| Detection | Browser `Accept-Language` on first visit, then `NEXT_LOCALE` cookie | Standard `next-intl` middleware behavior. |
| Switcher placement | New `LanguageMenu` component beside `ThemeToggle` in `StatusBar` | Symmetrical UX with theme switch. |
| Translation authoring | Claude writes all 6 locales directly in JSON | User chose option A; trades human review for speed. |
| Font loading for CJK | Conditionally load `Noto Sans SC` (zh) and `Noto Sans KR` (ko) only when locale matches | Avoid penalising other locales with extra font weight. |

## Architecture

### File layout

```
src/
├── i18n/
│   ├── routing.ts                # locales list, defaultLocale, localePrefix config
│   ├── request.ts                # getRequestConfig — server-side locale resolution
│   └── messages/
│       ├── en.json               # source of truth
│       ├── fr.json
│       ├── es.json
│       ├── de.json
│       ├── zh.json
│       └── ko.json
├── middleware.ts                 # createMiddleware from next-intl
└── app/
    └── [locale]/                 # ALL routes move here
        ├── layout.tsx            # root layout, sets <html lang>, NextIntlClientProvider
        ├── (public)/
        │   ├── layout.tsx
        │   └── page.tsx
        ├── playground/
        │   ├── layout.tsx
        │   └── <11 mini-app dirs>
        ├── love-timer/page.tsx
        └── not-found.tsx
```

The current `src/app/layout.tsx`, `src/app/(public)/`, `src/app/playground/`, `src/app/love-timer/`, `src/app/not-found.tsx` all move under `src/app/[locale]/`. The root `src/app/layout.tsx` is removed (Next.js will use the one inside `[locale]/`).

### Middleware

`src/middleware.ts` uses `createMiddleware` from `next-intl/middleware` with:
- `locales: ['en', 'fr', 'es', 'de', 'zh', 'ko']`
- `defaultLocale: 'en'`
- `localePrefix: 'as-needed'` (no prefix on default `en`, prefix on others)
- `localeDetection: true` (reads `Accept-Language` then writes `NEXT_LOCALE` cookie)

The `matcher` pattern excludes `/api`, `/_next`, `/assets`, `/favicon.svg`, and other static files.

## Components

### `LanguageMenu` (new)

Location: `src/components/ui/LanguageMenu.tsx`. Inserted in `StatusBar` immediately to the left or right of `ThemeToggle` (final position to be confirmed visually during implementation).

Behavior:
- **Trigger button**: globe SVG icon + 2-letter locale code in caps (e.g. `EN`). Same minimal styling as `ThemeToggle` — `p-2 text-text-secondary hover:text-gold transition-colors`.
- **Dropdown**: anchored under trigger, opens on click, closes on outside click / `Escape` / route change.
- **Items**: 6 rows, each showing flag emoji + native language name. The active locale row shows a gold check (✓) on the right.
  ```
  🇬🇧 English      ✓
  🇫🇷 Français
  🇪🇸 Español
  🇩🇪 Deutsch
  🇨🇳 中文
  🇰🇷 한국어
  ```
- **Animation**: Framer Motion fade + scale (matches existing UI motion language).
- **Style**: `VoidPanel`-inspired — dark surface, gold-dim border, subtle shadow.
- **A11y**: `aria-haspopup="menu"`, `aria-expanded`, focus trap inside menu when open, arrow-key navigation between items, `aria-current="true"` on active locale.

Click handler: uses `useRouter` + `usePathname` from `next-intl/navigation` to call `router.replace(pathname, { locale })`. The next-intl router preserves the rest of the URL (e.g. `/playground/snake` → `/fr/playground/snake`).

### `StatusBar` (modified)

Add `LanguageMenu` next to `ThemeToggle`. No other behavior changes. Both buttons remain right-aligned, same vertical alignment, separator dot or small gap consistent with the rest of the bar.

### `ThemeToggle` (unchanged)

No code change. Visual layout adjusted only via the parent `StatusBar`.

### Section components (Hero, About, Skills, Services, Projects, Contact, Playground)

Each replaces hardcoded English strings with `useTranslations('<section>')` calls. Component structure, styling, animations are unchanged. Example (`Hero.tsx`):

```tsx
const t = useTranslations('hero');
// ...
<SectionLabel>{t('label')}</SectionLabel>
// ...
<Button onClick={...}>{t.rich('viewWork') /* or t('viewWork') */}</Button>
```

For strings with markup (e.g. `"Bridging the gap between <strong>looks</strong> and <strong>works</strong>"`) use `t.rich()`.

### Playground mini-apps

Each of the 11 mini-app `page.tsx` files gets a translation namespace under `playground.<appName>`. Strings extracted: titles, button labels, status messages, instructions, score/timer captions, "Game Over" / "Restart" / "Pause" type text. Numeric/symbolic content stays as-is.

### `love-timer`

Same pattern: `useTranslations('loveTimer')`. All copy moves to JSON.

### `not-found`

Translated via `useTranslations('notFound')`.

### Metadata

`generateMetadata({ params: { locale } })` in each route's `page.tsx` (or the `[locale]/layout.tsx` root) reads from `getTranslations({ locale, namespace: 'metadata.<page>' })` to produce locale-aware `<title>`, `<meta description>`, OG tags. The OG `locale` field is set per request (`en_US`, `fr_FR`, `es_ES`, `de_DE`, `zh_CN`, `ko_KR`).

`<html lang={locale}>` is set in `[locale]/layout.tsx`.

`alternates.languages` map is added in metadata so search engines see hreflang variants.

### Sitemap

`src/app/sitemap.ts` (new file) emits one entry per (route × locale) pair with `alternates.languages` populated. Robots / Next.js handle the rest.

## Data flow

1. Browser hits `/some/path`.
2. `middleware.ts` runs. If path has no locale prefix and no cookie:
   - Reads `Accept-Language`, picks best match from supported locales.
   - If best match is `en` (default), serves the path as-is and sets `NEXT_LOCALE=en` cookie.
   - Otherwise, redirects to `/<locale>/some/path` and sets cookie.
3. Next.js routes the request to `app/[locale]/...` with `params.locale` populated.
4. `[locale]/layout.tsx` calls `unstable_setRequestLocale(locale)` and wraps children in `NextIntlClientProvider` with the loaded messages.
5. Server Components call `getTranslations({ namespace })`. Client Components call `useTranslations(namespace)`.
6. User clicks `LanguageMenu` → `LanguageX`. The next-intl client router pushes `/<newLocale>/<currentPathWithoutLocale>` and sets `NEXT_LOCALE=<newLocale>` cookie.

## Font loading for CJK

In `[locale]/layout.tsx`, conditionally inject the right Google Fonts link:
- `zh` → `Noto Sans SC` (and Display equivalent if available)
- `ko` → `Noto Sans KR`
- other locales → keep current font stack only

This uses `next/font/google` per-locale and a CSS variable swap in the body class so existing `font-display` utilities continue to work without per-component changes. CJK fonts are NOT loaded for non-CJK locales.

## Error handling

- **Missing translation key in a non-en file**: `next-intl` throws in dev, falls back to the key string in prod. We add a build-time check (script) that fails CI if any locale file is missing keys present in `en.json`.
- **Unsupported locale in URL** (e.g. `/jp/`): middleware returns 404 via `notFound()` route, which renders the localized `not-found.tsx` for the detected fallback.
- **Cookie tampering / unknown locale**: middleware ignores the cookie and re-runs detection.
- **Hydration mismatch on `<html lang>`**: avoided because the `[locale]` segment is server-resolved before render; no `suppressHydrationWarning` needed beyond the existing one for the theme.

## Testing

Pragmatic, no over-engineering:

- **Type safety**: `next-intl`'s `Messages` type augmentation typechecks every `t('key')` call at compile time. `tsc --noEmit` covers this.
- **Key parity script** (`scripts/check-i18n-parity.ts`): walks every locale file and asserts the key set matches `en.json` exactly. Fails with a diff. Wired into `npm run lint` (or as a separate `npm run check:i18n`).
- **Manual smoke checklist** before merge:
  - `/`, `/fr/`, `/es/`, `/de/`, `/zh/`, `/ko/` all render the homepage in the right language
  - Theme toggle still works on every locale
  - Language menu cycles correctly and preserves the current path (e.g. switching language while on `/playground/snake` lands on `/<newLocale>/playground/snake`)
  - CJK fonts render correctly on `zh` and `ko`
  - Lighthouse SEO check on the homepage in at least 2 locales (hreflang present)
- **No automated unit tests** for translations themselves — JSON is data, not logic. Translation correctness is a human review concern.

## Migration sequence (for plan phase)

The implementation plan will sequence:

1. Install deps, set up `routing.ts`, `request.ts`, `middleware.ts`.
2. Move `app/*` routes under `app/[locale]/`. Verify the site still builds and renders in English with no other changes (regression-safe checkpoint).
3. Build `LanguageMenu` and integrate into `StatusBar`.
4. Extract strings from layout components (`StatusBar`, `ChapterBar`, `Footer`, `HUDFrame`).
5. Extract strings from main sections (`Hero`, `About`, `Skills`, `Services`, `Projects`, `Contact`, `Playground` intro).
6. Extract strings from each playground mini-app (one at a time, each its own checkpoint).
7. Extract strings from `love-timer` and `not-found`.
8. Wire metadata + sitemap + hreflang per-locale.
9. Add CJK font loading.
10. Run key-parity script, manual smoke check, fix.

Each step ends with the site still buildable and visually unchanged in English (only the underlying mechanism changes). New locales become visible only once their JSON file is filled.

## Open questions / risks

- **Translation quality for ZH and KO**: Claude can produce grammatically correct CJK, but tone calibration ("technical, sober, premium") is harder to verify without a native reviewer. Risk accepted per user choice (option A). User can spot-fix later.
- **Visual layout in DE**: German strings are typically 30% longer than English. Hero name block, button labels, and stat captions need a quick visual pass on the DE locale to confirm no overflow. Mitigation: the layout is largely fluid (Tailwind clamp(), flex-wrap), but the dev-server smoke-check covers this.
- **CJK line-breaking** in long descriptions: CSS `word-break: keep-all` may be needed on some paragraph blocks for ZH/KO to avoid mid-word breaks. Address case-by-case during smoke check.
- **`Bariteau.` in Hero**: the gold-shimmer dot is part of the typography treatment. Name stays untranslated; only the surrounding tagline / description change.
